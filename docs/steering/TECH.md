## Architecture

### Pattern: Feature-based Vertical Slices

Each feature is a self-contained folder in `Web.Server/Features/` containing its own Controller, Service (with interface), and Models. No shared "Services" layer — business logic stays co-located with its API endpoint.

### Module structure

```
InterneTaakAfhandeling.sln
├── InterneTaakAfhandeling.Web.Server    → ASP.NET Core API + SPA host
│   ├── Features/{Feature}/              → Vertical slices (Controller + Service + Model)
│   ├── Config/                          → DI registration, auth policies
│   └── Database/                        → EF Core context (Kanalen only)
├── InterneTaakAfhandeling.Web.Client    → Vue 3 SPA (esproj)
│   ├── src/views/                       → Page components
│   ├── src/components/                  → Reusable components
│   ├── src/services/                    → API service modules
│   ├── src/stores/                      → Pinia state stores
│   └── src/composables/                 → Shared composition functions
├── InterneTaakAfhandeling.Common        → Shared API clients (OpenKlant, Objecten, Zaken, Email)
├── InterneTaakAfhandeling.Poller        → CronJob console app for notification polling
├── InterneTaakAfhandeling.EndToEndTest  → Playwright E2E tests
└── EndToEndTest.Common                  → Shared E2E test infrastructure
```

### Data flow

1. **Client → Server:** Vue SPA calls REST endpoints via `fetchWrapper.ts` (proxied in dev via Vite)
2. **Server → External APIs:** ASP.NET controllers delegate to feature services, which call API clients in `Common`
3. **Source of truth:** OpenKlant API and Objecten API — ITA only persists Kanalen and poller state locally in PostgreSQL
4. **Notifications:** Poller (K8s CronJob, every 15 min) checks for new internetaken and sends email via SMTP

### Authentication

- OpenID Connect via ASP.NET Core middleware
- Two authorization policies: `ITAPolicy` (basic access) and `FunctioneelBeheerderPolicy` (admin)
- Role claims mapped from identity provider (e.g. Azure Entra ID)

## Key Constraints

1. **OpenKlant API is the source of truth** — ITA never duplicates domain data locally. The only local persistence is Kanalen configuration and poller timestamp state.
2. **No vendor-specific logic** — ZGW communication to e-Suite goes through the podiumd-adapter. No e-Suite-specific code in ITA.
3. **Feature-based vertical slices** — Each feature owns its Controller + Service + Model in a dedicated folder. No shared "Services" layer.
4. **NL Design System theming only** — UI uses Utrecht component library and Den Haag design tokens. No custom CSS outside the design system.
5. **Nullable enabled, primary constructors preferred** — Enforced via `.editorconfig` across all projects.
6. **Interface-driven services** — All feature services implement an interface (`I{Feature}Service`) for testability and DI registration.

## Code Conventions

### C# (server)

- **Naming:** PascalCase for types and public members, `_camelCase` for instance fields, `s_camelCase` for static fields
- **Style:** Prefer `var`, block-scoped namespaces, braces on all control flow
- **Primary constructors:** Preferred for DI injection
- **Nullable:** Enabled project-wide — no suppression without justification
- **Feature structure:**
  - `{Feature}Controller.cs` — API endpoint (inherits `ControllerBase`)
  - `I{Feature}Service.cs` + `{Feature}Service.cs` — Business logic
  - `{Feature}Model.cs` — Request/response DTOs
- **Routing:** `[Route("api/{resource}")]`, lowercase URLs
- **DI registration:** All services in `Config/ServiceCollectionExtensions.cs` as `AddScoped`

### TypeScript / Vue (client)

- **Components:** PascalCase `.vue` files (e.g., `ContactverzoekDetails.vue`)
- **Composables:** `use-{name}.ts` (kebab-case, e.g., `use-loader.ts`)
- **Services:** `{name}Service.ts` (camelCase, e.g., `internetakenService.ts`)
- **Types:** `{name}.ts` in type folders
- **Stores:** Pinia stores in `src/stores/`
- **ESLint:** Vue essential + TypeScript recommended (flat config)
- **Prettier:** Configured via `.prettierrc.json`
- **Imports:** Absolute from `src/` via Vite alias

## Architecture Decision Records

| 4 | e-Suite via podiumd-adapter only | No vendor-specific logic in ITA; adapter translates ZGW API calls to e-Suite protocol | — |
| 5 | Show max 2 phone + 1 email for digitale adressen | Uses `omschrijving` field for phone labels; keeps detail view clean | — |
| 6 | Closed contactverzoek = status `verwerkt` | Removed from open lists but remains accessible via direct URL; no hard delete | — |
| 7 | First-page-only workaround (100 items) | OpenKlant API pagination limitation; temporary — only first page of Actor's contactverzoeken shown | — |

Full ADR documents: `docs/decision-record/`

## Known Pain Points

- **No unit test coverage** — Only Playwright E2E tests exist. New features rely on E2E for verification.
- **Pagination ceiling (ADR #7)** — Users with >100 assigned contactverzoeken see only the first page. Temporary workaround pending API improvement.
- **Werkverdeling not yet implemented** — Feature folder exists but is empty; upcoming epic will fill this.
