## Architecture

### Pattern: Feature-based Vertical Slices

Each feature is a self-contained folder in `Web.Server/Features/` containing its own Controller, Service (with interface), and Models. No shared "Services" layer — business logic stays co-located with its API endpoint.

### Module structure

```
InterneTaakAfhandeling.sln
├── InterneTaakAfhandeling.Web.Server    → ASP.NET Core API + SPA host
│   ├── Features/{Feature}/              → Vertical slices (Controller + Service + Model)
│   ├── Config/                          → DI registration, auth policies
│   └── Database/                        → EF Core context (Kanalen only)
├── InterneTaakAfhandeling.Web.Client    → Vue 3 SPA (esproj)
│   ├── src/views/                       → Page components
│   ├── src/components/                  → Reusable components
│   ├── src/services/                    → API service modules
│   ├── src/stores/                      → Pinia state stores
│   └── src/composables/                 → Shared composition functions
├── InterneTaakAfhandeling.Common        → Shared API clients (OpenKlant, Objecten, Zaken, Email)
├── InterneTaakAfhandeling.Poller        → CronJob console app for notification polling
├── InterneTaakAfhandeling.EndToEndTest  → Playwright E2E tests
└── EndToEndTest.Common                  → Shared E2E test infrastructure
```

### Data flow

1. **Client → Server:** Vue SPA calls REST endpoints via `fetchWrapper.ts` (proxied in dev via Vite)
2. **Server → External APIs:** ASP.NET controllers delegate to feature services, which call API clients in `Common`
3. **Source of truth:** OpenKlant API and Objecten API — ITA only persists Kanalen and poller state locally in PostgreSQL
4. **Notifications:** Poller (K8s CronJob, every 15 min) checks for new internetaken and sends email via SMTP

### Authentication

- OpenID Connect via ASP.NET Core middleware
- Two authorization policies: `ITAPolicy` (basic access) and `FunctioneelBeheerderPolicy` (admin)
- Role claims mapped from identity provider (e.g. Azure Entra ID)

## Key Constraints

1. **OpenKlant API is the source of truth** — ITA never duplicates domain data locally. The only local persistence is Kanalen configuration and poller timestamp state.
2. **No vendor-specific logic** — ZGW communication to e-Suite goes through the podiumd-adapter. No e-Suite-specific code in ITA.
3. **Feature-based vertical slices** — Each feature owns its Controller + Service + Model in a dedicated folder. No shared "Services" layer.
4. **NL Design System theming only** — UI uses Utrecht component library and Den Haag design tokens. No custom CSS outside the design system.
5. **Nullable enabled, primary constructors preferred** — Enforced via `.editorconfig` across all projects.
6. **Interface-driven services** — All feature services implement an interface (`I{Feature}Service`) for testability and DI registration.

## Code Conventions

### C# (server)

- **Naming:** PascalCase for types and public members, `_camelCase` for instance fields, `s_camelCase` for static fields
- **Style:** Prefer `var`, block-scoped namespaces, braces on all control flow
- **Primary constructors:** Preferred for DI injection
- **Nullable:** Enabled project-wide — no suppression without justification
- **Feature structure:**
  - `{Feature}Controller.cs` — API endpoint (inherits `ControllerBase`)
  - `I{Feature}Service.cs` + `{Feature}Service.cs` — Business logic
  - `{Feature}Model.cs` — Request/response DTOs
- **Controller vs Service responsibility:** In a vertical slice architecture each feature decides what fits best. As a general guideline: services own domain logic (fetching, orchestrating, validating), while presentation mapping (e.g. `ToResponse`-style DTO transformations) often fits naturally in the controller. Neither placement is wrong — choose what keeps the feature simple and readable.
- **Routing:** `[Route("api/{resource}")]`, lowercase URLs
- **DI registration:** All services in `Config/ServiceCollectionExtensions.cs` as `AddScoped`

### TypeScript / Vue (client)

- **Components:** PascalCase `.vue` files (e.g., `ContactverzoekDetails.vue`)
- **Composables:** `use-{name}.ts` (kebab-case, e.g., `use-loader.ts`)
- **Services:** `{name}Service.ts` (camelCase, e.g., `internetakenService.ts`)
- **Types:** `{name}.ts` in type folders
- **Stores:** Pinia stores in `src/stores/`
- **ESLint:** Vue essential + TypeScript recommended (flat config)
- **Prettier:** Configured via `.prettierrc.json`
- **Imports:** Absolute from `src/` via Vite alias

## Known Pain Points

- **No unit test coverage** — Only Playwright E2E tests exist. New features rely on E2E for verification.
- **Pagination ceiling (ADR #7)** — Users with >100 assigned contactverzoeken see only the first page. Temporary workaround pending API improvement.
- **Werkverdeling not yet implemented** — Feature folder exists but is empty; upcoming epic will fill this.

## Frontend Consistency & Accessibility

### Design System

The Vue 3 frontend MUST use the **Utrecht** (`@utrecht/component-library-vue`) and **Den Haag** (`@gemeente-denhaag/*`) design system component libraries as the primary UI building blocks. Custom styling MUST NOT duplicate or override design-token behavior.

- All UI primitives (buttons, headings, links, tables, form controls) MUST come from the NL Design System component libraries.
- Design tokens are imported via `@utrecht/design-tokens` and `@gemeente-denhaag/design-tokens-components`; project-specific overrides live in `src/assets/design-tokens.scss`.
- Custom SCSS MUST only use design-token variables — hard-coded colors, spacing, or typography values are prohibited.

> **Reference:** [NL Design System](https://nldesignsystem.nl/) · [Utrecht component library](https://nl-design-system.github.io/utrecht/) · [Den Haag design tokens](https://github.com/nl-design-system/denhaag)

### WCAG 2.2 & Digitoegankelijkheid Compliance

ITA is a Dutch government application and MUST comply with **WCAG 2.2 level AA** and the [Digitoegankelijk](https://digitoegankelijk.nl/) standard (Dutch implementation of the European EN 301 549 directive).

| Requirement | Enforcement |
| --- | --- |
| Semantic HTML | Use `<utrecht-heading>`, `<utrecht-paragraph>`, `<utrecht-link>` etc. — no `<div>` soup |
| Keyboard navigable | All interactive elements MUST be reachable and operable by keyboard alone |
| Focus management | Custom focus rings MUST use design-token `--utrecht-focus-*` values; never suppress `:focus-visible` |
| Color contrast | Text MUST meet ≥ 4.5:1 ratio (AA normal text); rely on design-token palette which is pre-validated |
| Form labels | Every form control MUST have an accessible `<label>` or `aria-label`; error messages MUST be linked via `aria-describedby` |
| Landmarks & headings | Pages MUST have a logical heading hierarchy (h1 → h2 → h3) and use ARIA landmarks (`<main>`, `<nav>`, `<aside>`) |
| Screen-reader announcements | Dynamic content changes (toasts, loading states) MUST use `aria-live` regions |
| Language attribute | `<html lang="nl">` MUST be set; mixed-language content MUST use inline `lang` attributes |

> **Reference:** [WCAG 2.2 Quick Reference](https://www.w3.org/WAI/WCAG22/quickref/) · [Digitoegankelijk.nl](https://digitoegankelijk.nl/) · [EN 301 549](https://www.etsi.org/deliver/etsi_en/301500_301599/301549/)

### Frontend Architecture Rules

| Rule | Description |
| --- | --- |
| Views | All new views MUST be placed in `src/views/` with descriptive `*View.vue` naming |
| Components | Reusable UI elements MUST be placed in `src/components/` |
| Features | Feature-specific logic MUST be scoped under `src/features/` |
| State management | State MUST use Pinia stores in `src/stores/` |
| API calls | MUST use the `fetchWrapper` utility in `src/utils/fetchWrapper.ts`; direct `fetch` calls are prohibited in components and services |
| Services | Service modules in `src/services/` MUST expose typed functions that return `Promise<T>` with explicit TypeScript types from `src/types/` |
| Composables | MUST follow the `use-*.ts` naming convention in `src/composables/` |
| Pattern reuse | When a new view requires behavioral logic similar to an existing view (e.g., pagination, state caching), the new view MUST reuse existing composables rather than reimplementing equivalent logic. If the existing composable does not fit, extend it or create a new composable after evaluation |
| Refactor on touch | When adding to or modifying an existing component or service, evaluate whether the current structure is still appropriate — don't just follow the existing pattern. If the code has grown in complexity or responsibility since it was first written, consider whether a refactor would keep it clean and maintainable before adding more. Improvement is not limited to the lines being changed. |
| Route definitions | MUST include `meta.title` and `meta.requiresITAAccess` (or `meta.requiresAuth` / `meta.requiresFunctioneelBeheerderAccess`) for authorization enforcement and page-title rendering |

**Rationale:** Consistent use of NL Design System tokens ensures accessibility, government branding compliance (Digitoegankelijk), and a uniform user experience across all views.
